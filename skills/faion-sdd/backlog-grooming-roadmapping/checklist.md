# Backlog Grooming Checklist

Checklists for grooming sessions, roadmap reviews, and backlog health audits.

---

## Weekly Grooming Session

### Pre-Session Preparation

- [ ] Review items added since last grooming
- [ ] Check upcoming sprint capacity
- [ ] Identify items needing stakeholder input
- [ ] Prepare RICE scores for new items
- [ ] Pull usage data for prioritization decisions

### During Session

#### 1. New Ideas Review (10 min)

- [ ] Review each new idea/feature request
- [ ] Quick assessment: Is this worth exploring?
- [ ] Assign initial priority (P0-P3)
- [ ] Decision: Keep (move to validated) or Drop
- [ ] Document reason for dropped items

#### 2. Prioritization (20 min)

- [ ] Review top 20 backlog items
- [ ] Re-score items with new information
- [ ] Check if any P0s should be demoted
- [ ] Verify only 1-3 items are P0
- [ ] Update item order based on priority

#### 3. Refinement (20 min)

- [ ] Add acceptance criteria to next sprint items
- [ ] Break down items that are too large
- [ ] Identify and document dependencies
- [ ] Ensure items have clear "Definition of Done"
- [ ] Verify technical feasibility with dev team

#### 4. Cleanup (10 min)

- [ ] Archive items older than 6 months (not touched)
- [ ] Merge duplicate items
- [ ] Update complexity/effort estimates
- [ ] Move completed items to done/
- [ ] Update "Won't Do" list if applicable

### Post-Session

- [ ] Document grooming decisions
- [ ] Notify stakeholders of priority changes
- [ ] Update roadmap if needed
- [ ] Schedule follow-ups for blocked items

## Backlog Item Quality Checklist

### For Ideas (Pre-Validation)

- [ ] Has clear problem statement
- [ ] Identifies target user segment
- [ ] Has initial value hypothesis
- [ ] Not a duplicate of existing item

### For Validated Items

- [ ] Problem validated with data or user feedback
- [ ] Has measurable success criteria
- [ ] Aligns with product strategy
- [ ] Has RICE score calculated
- [ ] MoSCoW category assigned

### For Ready Items (Sprint-Ready)

- [ ] Has complete spec.md
- [ ] Has technical design.md
- [ ] Acceptance criteria are testable
- [ ] Dependencies identified and resolved
- [ ] Fits within 100k token context
- [ ] Has implementation plan with tasks
- [ ] Team understands requirements

## Roadmap Review Checklist

### Monthly Review

- [ ] Review "Now" items progress
- [ ] Assess blockers and risks
- [ ] Move completed items
- [ ] Reprioritize "Next" based on learnings
- [ ] Add new items from backlog
- [ ] Update stakeholder communication

### Quarterly Review

- [ ] Evaluate quarter outcomes vs goals
- [ ] Review theme/initiative progress
- [ ] Adjust "Next" and "Later" priorities
- [ ] Incorporate market/competitor changes
- [ ] Update resource allocations
- [ ] Communicate changes to stakeholders
- [ ] Archive completed themes
- [ ] Define next quarter's "Now" items

---

## Prioritization Decision Checklist

### Before Scoring (RICE)

- [ ] Reach: Have data on affected users?
- [ ] Impact: Clear on value per user?
- [ ] Confidence: Honest about uncertainty?
- [ ] Effort: Consulted with engineering?

### Before Categorizing (MoSCoW)

- [ ] Must: Would release fail without it?
- [ ] Should: Can we launch without it?
- [ ] Could: Is this truly optional?
- [ ] Won't: Documented for future reference?

### Before Finalizing Priority

- [ ] Compared to items at same priority level
- [ ] Checked for dependencies
- [ ] Verified resource availability
- [ ] Stakeholder alignment confirmed
- [ ] Decision documented with reasoning

## Backlog Health Audit (Quarterly)

### Size & Age

- [ ] Total items < 100 (manageable size)
- [ ] No items older than 12 months without review
- [ ] Stale items archived or dropped
- [ ] New items rate is sustainable

### Quality

- [ ] All items have clear problem statements
- [ ] Priority distribution reasonable (not all P0)
- [ ] Items are appropriately sized
- [ ] Acceptance criteria are testable

### Process

- [ ] Weekly grooming happening consistently
- [ ] Decisions are documented
- [ ] Stakeholders informed of changes
- [ ] Team capacity reflected in planning

### Alignment

- [ ] Items align with product strategy
- [ ] Roadmap reflects current priorities
- [ ] "Won't Do" list is maintained
- [ ] Technical debt is tracked

---

## Sprint Planning Readiness

### For Each Sprint Candidate

- [ ] Has clear user story format
- [ ] Acceptance criteria defined
- [ ] Effort estimated (story points or t-shirt size)
- [ ] Dependencies resolved or scheduled
- [ ] Technical approach agreed
- [ ] No open questions requiring stakeholder input
- [ ] Team agrees it fits in sprint capacity

### For Sprint as a Whole

- [ ] Total effort within team capacity
- [ ] Mix of item types (features, bugs, tech debt)
- [ ] Sprint goal is clear and achievable
- [ ] No more than 20% carry-over from previous sprint
- [ ] Critical dependencies with other teams identified

---

## Stakeholder Communication Checklist

### When Priority Changes

- [ ] Identify affected stakeholders
- [ ] Prepare rationale with data
- [ ] Communicate before they discover
- [ ] Document the change and reason
- [ ] Update roadmap artifacts

### When Items Are Dropped

- [ ] Document why item was dropped
- [ ] Add to "Won't Do" list with explanation
- [ ] Notify original requestor
- [ ] Archive item for future reference
- [ ] Update related documentation

### When Roadmap Updates

- [ ] Summarize what changed
- [ ] Explain why (new data, priorities, resources)
- [ ] Highlight new commitments
- [ ] Note removed/delayed items
- [ ] Provide next review date
