# Card Sorting

## Summary

**One-sentence:** Navigation categories that make sense internally often fail users because they mirror org charts or product taxonomies rather than user mental models.

**One-paragraph:** Navigation categories that make sense internally often fail users because they mirror org charts or product taxonomies rather than user mental models. Card sorting surfaces the actual groupings users expect before IA decisions are locked in. Run 15-20 participants for open sorts (where users create categories), 30+ for closed sorts (where users place content into predefined categories), use 30-60 cards written in user language, and analyze results via similarity matrix to identify high-confidence groupings and ambiguous items.

## Applies If (ALL must hold)

- Designing or redesigning primary navigation—validate category labels and groupings before implementation.
- When analytics show high search use and low navigation use—users cannot find content in the current IA.
- Before tree testing—generate the structure that tree testing will then validate.
- When migrating content to a new IA—ensure the new structure matches user expectations.
- When internal opinion on IA is divided—card sort results provide neutral, data-driven evidence.

## Skip If (ANY kills it)

- When the IA is already validated and stable—card sorting is generative; use tree testing to validate an existing structure instead.
- With fewer than 15 participants for open sort—below that threshold, patterns are unreliable and may reflect individual quirks rather than shared mental models.
- When cards exceed 60 items—cognitive fatigue degrades result quality; participants rush the final cards. Split oversized sets into multiple focused sorts.
- For navigation with fewer than 10 items—direct user interviews are faster and surface more detailed reasoning.
- When IA is constrained by legal or regulatory requirements that cannot change regardless of user preference.
