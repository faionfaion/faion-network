# Stakeholder Communication

## Summary

**One-sentence:** Stakeholder communication selects a dialogue mode before any conversation: Interview (requirements gathering), Brainstorm (option generation), Clarification (resolving ambiguity), Validation (confirming understanding), or Socratic (probing assumptions).

**One-paragraph:** Stakeholder communication selects a dialogue mode before any conversation: Interview (requirements gathering), Brainstorm (option generation), Clarification (resolving ambiguity), Validation (confirming understanding), or Socratic (probing assumptions). Selecting the wrong mode — e.g., using Socratic questions during a validation session — produces incoherent conversations and erodes trust. Stakeholder mapping by power/interest determines communication frequency and channel per quadrant.

## Applies If (ALL must hold)

- Preparing interview guides for discovery sessions (requirements gathering, user research, product feedback)
- Mapping stakeholders by power/interest before a project kickoff
- Drafting post-meeting validation summaries that stakeholders must confirm before work proceeds
- Generating Socratic question sequences to probe vague requirements like "make it faster" or "improve UX"
- Handling difficult stakeholder types (silent, dominant, negative, absent, changing)

## Skip If (ANY kills it)

- Replacing live dialogue — agent-generated questions are preparation aids, not substitutes for real conversations
- Situations with high political sensitivity where each word carries organizational weight beyond the stated content
- Legal or contractual negotiations requiring specific protocol
- Power/Interest grid placement without a human who knows the organizational context — job titles alone misplace politically influential stakeholders
