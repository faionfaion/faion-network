Given the following feature brief:
<brief>
{BRIEF}
</brief>

Produce a spec.md following these steps in order:

1. Write the Business Value Statement: [WHO] currently cannot [WHAT] because [WHY]. This results in [IMPACT]. By implementing [SOLUTION], we will [BENEFIT].

2. Define exactly 2 user personas. For each: name/archetype, role, goal, pain points, context (device/environment/when used).

3. Write user stories in "As a [persona], I want [action], so that [benefit]" format. Validate each against INVEST criteria — flag any that fail with a reason. Assign Must/Should/Could priority. Must stories must not exceed 60% of total stories.

4. Write FR-X functional requirements using "System SHALL..." format. One requirement per FR. No ambiguous words (fast, good, easy, support). Each FR: traces to US-X, has MoSCoW priority, includes validation rules if input is involved. Must FRs may not exceed 60% of total FR count.

5. Write NFR-X for performance, security, and availability. Each NFR must include: specific numeric target, measurement conditions (load, percentile), and measurement method.

6. Write AC-X acceptance criteria in Given-When-Then format for each Must FR. Use specific values, not placeholders. Cover: AC-001 happy path, AC-002 validation error, AC-003 boundary condition, AC-004 security scenario. Add AC-005 performance scenario if a Must NFR applies.

7. Write explicit In Scope and Out of Scope sections. Out of Scope must name specific features with reason and timeline.

8. Document assumptions and constraints.

Output: complete spec.md following the template structure. Do not invent performance numbers — use only values from the brief or state "TBD — requires NFR workshop".
