== SPEC CREATION PROMPT ===============================================

You are writing a specification for: {feature-name}

Input context:
- Business problem: [describe the pain point]
- Users affected: [who is impacted]
- Constraints from constitution.md: [paste relevant sections]
- Similar existing features: [list paths to similar specs]

RULES:
- Write requirements as verifiable behaviors, not implementation instructions
- Every FR-X must have at least one Given-When-Then acceptance criterion
- Separate what the system must do (FR) from how it will work (design.md)
- Non-goals must be explicit — list at least 3 things this spec does NOT cover
- Open questions must block work that cannot proceed without answers

OUTPUT: spec.md following the spec-full.md template.

Start by listing the top 3-5 functional requirements as one sentence each.
Then expand each into full FR sections with acceptance criteria.

== REQUIREMENTS REVIEW PROMPT =========================================

Review this spec for quality: [paste spec content]

Check each FR-X:
1. Is it verifiable? Can a test be written for it?
2. Is it specific? Does it name the exact behavior, not "should work well"?
3. Is it complete? Does it cover the error/edge case as well as the happy path?
4. Is it in scope? Does it match the problem statement?

Check the spec overall:
5. Non-goals: are there at least 3 explicit exclusions?
6. Open questions: are all unknowns that block design listed?
7. Success criteria: are they measurable (numbers, not adjectives)?
8. Scope creep: does any FR go beyond the stated problem?

Output: numbered list of issues with FR reference and fix suggestion.

== AC SHARPENING PROMPT ===============================================

These acceptance criteria are too vague: [paste AC]

Rewrite each one to be:
- Specific: names exact endpoint, field, or element
- Measurable: has a concrete result (status code, value, message text)
- Testable: can be verified by running code, not by human judgment

Bad example: "The system responds quickly"
Good example: "When POST /api/orders returns 201, the response body contains an order.id field of type string"

== NFR GENERATION PROMPT ==============================================

Based on this spec context: [paste problem statement + FR list]

Generate appropriate non-functional requirements:

1. Performance: what response time targets are reasonable for this feature type?
2. Reliability: what availability target fits the criticality?
3. Security: what auth, data handling, or compliance requirements apply?
4. Scalability: what load should this handle at launch vs. 12 months out?

For each NFR, provide a specific measurable target, not a vague statement.
