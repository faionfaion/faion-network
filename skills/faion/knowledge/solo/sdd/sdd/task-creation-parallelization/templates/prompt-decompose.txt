== DECOMPOSITION PROMPT ================================================

You are decomposing an approved design.md into executable TASK-XXX-*.md files.

RULES:
- Each task must fit within 100k tokens (implementation + context + tests)
- Each task touches one cohesive area: one module, one API layer, one data model
- Tasks within the same wave MUST NOT write to the same file (parallel git worktrees)
- Every task must have at least one Given-When-Then acceptance criterion
- Every task must trace to at least one FR-X from spec.md

INPUT:
- spec.md: [paste spec content here]
- design.md: [paste design content here]

OUTPUT:
For each task, produce a TASK-XXX-{title}.md file following task-file.md template.
At the end, produce a task-list.md following task-list.md template.

START with the dependency analysis:
1. List all logical units of work (modules, layers, integrations)
2. Identify which units can proceed in parallel (no shared file writes)
3. Identify which units have strict finish-to-start dependencies
4. Assign wave numbers: Wave 1 = no deps, Wave 2 = depends on Wave 1 only, etc.

== WAVE PLANNING PROMPT ===============================================

Given these tasks and dependencies:
[paste task list with dependencies]

Plan the wave execution:
1. Assign each task to the earliest wave where all dependencies are DONE
2. Flag any tasks that CANNOT be parallelized (shared file writes) — move to sequential
3. Estimate total context-token load per wave
4. Identify the critical path (longest chain of dependencies)

Output: updated task-list.md with wave assignments

== AC GENERATION PROMPT ===============================================

For task: [paste task description and FR references]

Generate 2-3 acceptance criteria in Given-When-Then format:
- AC-1: Happy path (what works when everything is correct)
- AC-2: Error/edge case (what happens when preconditions fail)
- AC-3 (optional): Performance or security boundary

Each AC must be:
- Verifiable by running code (not "should feel responsive")
- Specific (names the exact function, endpoint, or UI element)
- Independent of implementation details

== TASK VERIFICATION PROMPT ===========================================

Review this task file: [paste TASK-XXX content]

Check:
1. Token estimate: does the sum of SDD Docs + Dependency Tree + Research + Implementation + Testing stay under 100k?
2. INVEST: is the task Independent within its wave? Valuable (traces to FR-X)? Testable (has AC)?
3. File conflicts: does it write to any file that another same-wave task also writes?
4. Dependency completeness: is every input this task needs listed in Depends On?
5. AC completeness: does each FR referenced have at least one verifiable AC?

Report any violations; suggest fixes.
