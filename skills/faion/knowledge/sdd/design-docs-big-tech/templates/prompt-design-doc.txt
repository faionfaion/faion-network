== DESIGN DOC CREATION PROMPT =========================================

You are writing a technical design document for: {feature-name}

Context:
- spec.md: [paste spec content]
- constitution.md excerpts: [paste relevant sections]
- Existing similar designs: [list paths]

RULES:
- One architectural decision per AD-X section (not "we use A and B and C")
- Every AD-X must include: Options Considered, Decision, Rationale, Trade-offs
- API contracts in spec format (types, not prose)
- Trace every component to a FR-X requirement
- Flag open questions explicitly in an "Open Questions" section

OUTPUT STRUCTURE:
1. Overview (2-3 paragraphs: problem, solution, scope)
2. Architecture (component diagram as text, data flow)
3. Component Design (one section per major component)
4. API Contracts (request/response shapes, error codes)
5. Data Models (schema fields, types, constraints)
6. Architecture Decisions (AD-1 through AD-N)
7. Open Questions (unresolved issues blocking execution)
8. Out of Scope (explicit exclusions)

== AD-X DECISION PROMPT ===============================================

Write an Architecture Decision for: {decision topic}

Options considered:
- Option A: {name} — {one-line description}
- Option B: {name} — {one-line description}
- Option C: {name} — {one-line description}

For each option provide:
- Pros: {what it does well}
- Cons: {what it does poorly}
- When to choose: {conditions where this is the right pick}

Then write the decision section:
- Decision: Option {X} — {rationale in 2-3 sentences}
- Trade-offs accepted: {what we give up}
- Rejected because: {why Options Y and Z were not chosen}

== DESIGN REVIEW PROMPT ===============================================

Review this design document: [paste design content]

Check for:
1. Completeness: every FR-X from spec.md has a corresponding design section
2. Precision: API contracts include types (not just field names)
3. Decision quality: every AD-X has a stated rationale (not just "we chose X")
4. Conflict detection: do any AD-X decisions contradict constitution.md?
5. Open questions: are all blockers for execution listed?
6. Scope creep: does the design cover anything NOT in the spec?

Output: numbered list of issues with severity (blocking / advisory)
