== DESIGN DOC KICKOFF PROMPT ==========================================

You are writing a design document for: {feature-name}

Input:
- spec.md: [paste spec content]
- constitution.md (relevant sections): [paste]
- Existing similar designs (for pattern reference): [list paths]

RULES:
- One architectural decision per AD-X section
- Every AD-X must include: Options Considered, Decision, Rationale, Trade-offs
- API contracts must be in typed spec format (not prose)
- Every component must trace to at least one FR-X
- Open Questions must list only genuine blockers — not research items
- Out of Scope must have at least 2 explicit exclusions

Start by listing the top 3-5 architectural decisions that need to be made.
Then draft the overview and architecture sections.
Then expand each decision into a full AD-X section.

== AD-X GENERATION PROMPT =============================================

Write architecture decision AD-{N} for: {decision topic}

Context:
- Constraint from constitution.md: [paste]
- Requirements from spec.md FR-{X}: [paste]
- Existing codebase patterns: [paste]

Generate:
1. Three options with honest pros/cons
2. A clear decision statement
3. Rationale referencing the constraints
4. Trade-offs explicitly accepted

Format as AD-{N} section from design-sdd.md template.

== DESIGN REVIEW PROMPT ===============================================

Review this design document for quality: [paste design content]

Check:
1. FR coverage: does every FR-X from spec.md have a corresponding design component?
2. Decision quality: does every AD-X have a stated rationale (not just "we chose X")?
3. Conflict detection: do any AD-X decisions contradict constitution.md?
4. API precision: are all request/response shapes typed (not just named)?
5. Open questions: are all genuine blockers listed?
6. Scope: does the design cover anything not in the spec?
7. Testability: can every component be tested in isolation?

Output: numbered list of issues with severity — BLOCKING / ADVISORY / QUESTION

== LLM CODE GENERATION CONTEXT PROMPT ================================

Generate the context block for LLM code generation from this design: [paste design content]

The context block should include:
1. Component being implemented (name, responsibility)
2. Inputs and outputs (types, not just names)
3. Architecture decisions that constrain this component (AD-X references + key rule)
4. Naming conventions from constitution.md
5. Error handling pattern to follow
6. Path to a similar existing implementation
7. Tests required (from AC in task file)

Output: a self-contained context block that can be pasted into a code generation prompt.
Target: < 3k tokens (so it fits with implementation space to spare in 100k context).

== VALIDATE DESIGN PROMPT =============================================

Validate that this design is ready for task decomposition: [paste design content]

Gate checklist:
- [ ] Every FR-X from spec.md has a design section
- [ ] Every AD-X has Options Considered, Decision, Rationale, Trade-offs
- [ ] All API contracts have typed request/response shapes
- [ ] Open Questions list is empty or each item has an owner and deadline
- [ ] Out of Scope section has at least 2 entries
- [ ] No component is described as "TBD" or "to be determined"

Output: APPROVED (all checks pass) or BLOCKED (list failing checks with what's needed)
