== FAST REVIEW PROMPT (pre-commit, ~2 min) ============================

You are reviewing code changes for a quick correctness and safety check.

Changed files: [paste git diff or file list]
Task being completed: [paste TASK-XXX title and AC]

Check only:
1. Logic errors that would cause the AC to fail
2. Security: secrets in code, SQL injection, unvalidated input
3. Missing error handling on I/O operations (file, network, DB)
4. Breaking changes to public interfaces not covered by the task

Output: PASS or list of BLOCKING issues (max 5 lines each). No style feedback.

== THOROUGH REVIEW PROMPT (post-implementation, ~10 min) ==============

You are performing a thorough review of: [feature or module name]

Context:
- spec.md: [paste relevant FR sections]
- design.md: [paste relevant AD-X decisions]
- Changed files: [paste git diff]
- Existing patterns (from patterns.md): [paste relevant patterns]

Review dimensions:

1. CORRECTNESS
   - Does the code implement every FR-X in the spec?
   - Does it handle all error cases listed in the AC?
   - Are edge cases (empty input, max values, concurrent access) handled?

2. DESIGN CONSISTENCY
   - Does naming follow the conventions in constitution.md?
   - Does the implementation match the architecture decisions (AD-X)?
   - Are new patterns consistent with existing ones in the codebase?

3. SECURITY
   - Authentication and authorization: is every endpoint protected?
   - Input validation: is user input sanitized before use?
   - Secrets: no hardcoded tokens, passwords, or keys
   - Dependencies: no new deps with known CVEs

4. TESTABILITY
   - Are the acceptance criteria from TASK-XXX verifiable by running tests?
   - Are tests isolated (no shared state between test cases)?
   - Are error paths tested, not just happy paths?

5. MAINTAINABILITY
   - Is complex logic explained with comments?
   - Are functions small enough to understand in one reading (< 40 lines)?
   - Are there TODO/FIXME comments that should be issues instead?

Output format:
- BLOCKING: {issue} — must fix before merge
- ADVISORY: {issue} — should fix, can follow-up
- QUESTION: {clarification needed}
- PRAISE: {what was done well — helps establish patterns}

== SPEC COMPLIANCE CHECK ==============================================

Verify this implementation against its spec: [paste spec FR list]

For each FR-X:
- Find the code that implements it
- Verify the AC (Given-When-Then) is satisfied
- Note any FR that has no corresponding implementation
- Note any implementation that has no corresponding FR (scope creep)

Output:
| FR | Implemented | AC Verified | Notes |
|----|------------|-------------|-------|
| FR-1 | yes/no/partial | yes/no | {comment} |

== REFLEXION PROMPT (after review cycle) ==============================

After completing this review cycle, capture what was learned.

Review outcome:
- Issues found: [list issues]
- Issues that were preventable: [which ones]
- Root cause: [why did these issues occur?]

Generate:
1. Pattern entry (if a good pattern was confirmed or established)
2. Mistake entry (if a recurring error pattern was found)
3. Constitution amendment (if a rule should be added to prevent recurrence)

Format for patterns.md:
```
## {pattern-name}
**Confidence:** Low/Medium/High
**Context:** {when to apply}
**Pattern:** {what to do}
**Example:** {code path that uses this}
```

Format for mistakes.md:
```
## {mistake-name}
**Frequency:** {how often seen}
**Root cause:** {why it happens}
**Fix:** {how to avoid}
**Detection:** {how to catch in review}
```
