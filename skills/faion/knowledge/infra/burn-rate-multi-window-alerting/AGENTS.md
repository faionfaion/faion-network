# Burn-rate Multi-window Alerting

## Summary

**One-sentence:** Canonical Google-SRE 1h/6h/24h burn-rate alert recipe (page + ticket levels) wired to a published SLO and reviewed quarterly.

**One-paragraph:** Canonical Google-SRE 1h/6h/24h burn-rate alert recipe (page + ticket levels) wired to a published SLO and reviewed quarterly. The methodology pins the discipline that turns folklore into a reviewable, owned, version-controlled operating artefact: rule-bound output contract, evidence anchors, named owner, published review cadence. Outputs of the wrong shape are rejected at review; outputs without evidence are demoted to hypotheses; outputs without owners are tagged stale.

## Applies If (ALL must hold)

- Service has a published SLO with an explicit error budget.
- Metric source exposes good + bad event counts via Prometheus-compatible scrape.
- On-call rotation is in place and a paging vs ticket distinction is meaningful.

## Skip If (ANY kills it)

- No SLO is defined — write the SLO first.
- Error budget is effectively zero (regulatory 100% — burn-rate math is meaningless).
- Service has < 100 events/minute (sample size too small for burn-rate signal).

**Ефективно для:**

- Сервіси з SLO ≥ 99.9% і error-budget > 0.
- Алерти на основі Prometheus / Mimir / Grafana Cloud.
- Команди де page-fatigue знижує MTTR.
- Регулярний quarterly SLO review.
