# ArgoCD GitOps Deployment Policy

## Summary

**One-sentence:** Pull-based GitOps deployment spec with ApplicationSets, sync waves, sync windows, server-side apply, and per-environment self-heal policy — keeps cluster and Git state convergent.

**One-paragraph:** Pull-based GitOps deployment spec with ApplicationSets, sync waves, sync windows, server-side apply, and per-environment self-heal policy — keeps cluster and Git state convergent. The methodology pins the discipline that turns folklore into a reviewable, owned, version-controlled operating artefact: rule-bound output contract, evidence anchors, named owner, published review cadence. Outputs of the wrong shape are rejected at review; outputs without evidence are demoted to hypotheses; outputs without owners are tagged stale.

## Applies If (ALL must hold)

- Cluster fleet ≥ 1 cluster, deploys ≥ once/week.
- Team accepts Git-as-source-of-truth model.
- Helm or Kustomize is the manifest format.

## Skip If (ANY kills it)

- Non-Kubernetes workloads (VMs, serverless).
- Team wants push-based CI kubectl deploys.
- Single-developer single-cluster project — overhead exceeds benefit.

**Ефективно для:**

- Kubernetes deploys на ≥ 2 cluster + multi-env.
- Команди де kubectl apply з CI створює drift.
- Audit-heavy організації які потребують Git як source-of-truth.
- Preview env per PR через ApplicationSet PR generator.
