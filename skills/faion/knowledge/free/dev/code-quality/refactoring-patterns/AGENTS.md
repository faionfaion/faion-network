# Refactoring Patterns

## Summary

**One-sentence:** A closed catalog of eight structure-preserving transforms — extract method/class, replace conditional with polymorphism, introduce parameter object, replace magic numbers, decompose conditional, rename for clarity, move method — applied one at a time behind a green test suite.

**One-paragraph:** A closed catalog of eight structure-preserving transforms — extract method/class, replace conditional with polymorphism, introduce parameter object, replace magic numbers, decompose conditional, rename for clarity, move method — applied one at a time behind a green test suite. Each refactor is a single commit. External behavior must remain identical after every step.

## Applies If (ALL must hold)

- Code works but is hard to understand or touch without fear.
- Adding a feature would require changes in many unrelated files.
- Duplicate logic exists across the codebase.
- A class or function has grown beyond a single clear responsibility.
- Before adding new functionality ("make the change easy, then make the easy change").
- Spreading a rename or API change across many call sites in a monorepo.

## Skip If (ANY kills it)

- No test coverage on the target code — refactoring without a safety net silently changes behavior.
- Cross-cutting architectural rewrite (monolith → services, CQRS introduction) — use software-architect patterns instead.
- Refactors require domain knowledge the agent lacks (regulatory rules, financial rounding, license-encoded behavior).
- During incidents or release-freeze windows — risk outweighs benefit.
- Code generated by another agent in the same session — re-editing for taste, not improvement.
