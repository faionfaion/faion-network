# Agent Integration — Refactoring Patterns

## When to use
- Mechanical refactors: extract method/class/parameter object, rename for clarity, replace magic numbers, decompose conditionals — high pattern density, low judgment.
- Pre-feature preparatory refactors ("make the change easy, then make the easy change") behind a strong test suite.
- Spreading a single refactor (e.g., introduce parameter object) across many call sites in a monorepo.
- Large-scale codemod / API migration where AST tools alone produce ugly results and need contextual cleanup.
- Translating between paradigms (conditional → polymorphism, inheritance → composition) where pattern recognition is the bottleneck.

## When NOT to use
- Refactoring without tests. Skip until tests exist; agent will silently change behavior.
- Cross-cutting architectural reshape (CQRS introduction, monolith→services). Use `software-architect` skill, not refactoring.
- Refactors that require domain knowledge the agent lacks (regulatory rules, financial calculations, license-encoded behavior).
- "Polish" refactors during incidents or release-freeze windows — risk > reward.
- Code generated by another agent in the same session — refactoring agent will rewrite for taste, not improvement.

## Where it fails / limitations
- Behavior preservation is unverified without a test run. Agent claims "no behavior change" while introducing subtle off-by-ones (rounding, datetime, encoding).
- Cross-file refactors miss callers in test files, scripts, docs, generated code.
- Pattern over-application: agent introduces Strategy classes for a 3-branch conditional; abstraction debt > original debt.
- Type inference: dynamic-typed languages (Python, Ruby, JS) make extract-method risky; agent often gets parameter types wrong.
- Concurrency-sensitive code (locks, async ordering) breaks under naive extract-method.
- Public-API renames need release-note + deprecation handling the agent forgets.

## Agentic workflow
Run as a tight TDD-style loop. Step 1: ensure tests exist for the target — `coverage-checker` agent reports gaps; pause if coverage <80% on touched lines. Step 2: `refactor-applier` agent applies ONE pattern at a time (extract method, then commit, then next). Step 3: full test suite runs after each commit. Step 4: human reviews diff before merge. Never batch multiple refactor patterns in one commit. Use the standard catalog in `README.md` as the closed list — agent must pick from it, not invent new patterns.

### Recommended subagents
- `coverage-checker` — reports per-line coverage on the target file/range; gates the loop.
- `refactor-planner` — reads code, picks ONE applicable pattern from the catalog, explains why, generates the diff. Sonnet/opus.
- `refactor-applier` — applies the diff via Edit tool, runs tests, commits if green.
- `rename-propagator` — handles cross-file rename via LSP/AST; non-LLM where possible.
- `simplify` (built-in skill) — review-changed-code pass; catches over-abstraction the planner introduced.
- `faion-sdd-executor-agent` — wraps the loop as an SDD task with pattern-list as plan.

### Prompt pattern
Planner:
```
Pick exactly ONE refactoring pattern from this catalog:
[extract-method, extract-class, replace-conditional-with-polymorphism,
 introduce-parameter-object, replace-magic-numbers, decompose-conditional,
 rename-for-clarity, move-method].

Apply it to: <code>
Constraints:
- Behavior preservation is mandatory. Cite which test(s) verify the affected code.
- Output: { pattern, justification, unified_diff, test_command }.
- If no catalog pattern fits cleanly, return { pattern: null, reason }.
```

Applier (after planner):
```
Apply the unified diff. Run <test_command>. If exit 0, commit with message
"refactor: <pattern> in <file>". If non-zero, revert and report the failing test.
```

## Templates & scripts
See `examples.md` for catalog with before/after pairs. Minimal "one pattern per commit" loop:

```bash
#!/usr/bin/env bash
# refactor-loop.sh — apply one pattern, test, commit, repeat
set -euo pipefail
TARGET="${1:?path required}"
TESTS="${2:?test cmd required}"
MAX="${3:-5}"

for i in $(seq 1 "$MAX"); do
  PLAN=$(claude -p "Plan ONE refactor on $TARGET. Output JSON {pattern, diff, test:'$TESTS'}." )
  PATTERN=$(jq -r .pattern <<<"$PLAN")
  [ "$PATTERN" = "null" ] && { echo "No more refactors."; break; }
  jq -r .diff <<<"$PLAN" | git apply --3way
  if eval "$TESTS"; then
    git add -A && git commit -m "refactor: $PATTERN in $TARGET"
  else
    git checkout -- . ; echo "Reverted: $PATTERN broke tests" ; break
  fi
done
```

## Best practices
- One pattern, one commit, one test run. Bisecting a regression demands clean history.
- Coverage gate first. <80% line coverage on touched code = stop and write tests, not refactor.
- Run mutation tests on critical modules before/after — they catch "tests pass but behavior changed".
- Forbid new abstraction without 2 concrete uses (rule of three). Agent will violate this without a hard prompt rule.
- Lock the catalog: closed list of patterns the agent may apply. Open-ended "improve" prompts produce taste-driven churn.
- Public APIs: never rename without a deprecation alias for one release cycle. Add a CI rule that blocks PRs renaming exported symbols without a `# DEPRECATED` shim.
- Pair refactor PRs with `BEFORE/AFTER` benchmarks for any code on a hot path. Cleanliness is not a perf license.

## AI-agent gotchas
- "Equivalent" rewrites lose subtle semantics: short-circuit eval order, exception types, decimal vs float, locale-dependent formatting. Demand verbatim test coverage on touched lines.
- Type-hint pollution: agent adds `Optional[str]` where None was already a contract violation, masking real bugs.
- Magic-number replacement renames `0.08` to `TAX_RATE` — fine, but agent tends to inline domain meaning into code that should read it from config. Constrain to in-file constants only.
- Polymorphism trap: 3-branch conditional → 4-class hierarchy. Force min-cases threshold (≥4 distinct branches) before allowing this pattern.
- Cross-repo callers: monorepo-wide renames miss tests in sibling packages, dynamically-imported strings, and YAML/JSON references. Use Sourcegraph batch changes or `git grep` validator.
- Decimal vs float in Python (sample shows `Decimal("0.08")`): agent freely converts between them; mixing yields TypeErrors at runtime not caught by lint.
- Snapshot tests: refactors break snapshot tests for cosmetic reasons (output ordering, whitespace). Distinguish behavior-preserving snapshot updates from regressions.
- Commit messages bloat: agent writes 200-word "what + why" essays. Constrain to 50-char title + ≤3-line body.
