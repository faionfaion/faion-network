# Personas

## Summary

**One-sentence:** Personas are fictional characters representing key user types, based on research.

**One-paragraph:** Personas are fictional characters representing key user types, based on research. They unify team decisions by answering "would our user want this?" without shared personas, teams design by assumption, team members disagree on who the user is, discussions become opinion battles, and features nobody wants ship. Cap personas at 3–5 primary; each must include goals, frustrations, behaviors, and at least one verbatim quote tied to a research participant. Pair every persona with a JTBD ("Sarah hires the dashboard to look prepared in meetings"). Keep persona docs short (≤1 page); long personas get skimmed and forgotten. Date-stamp personas; mark as DRAFT until validated against ≥10 users. Re-cluster after every meaningful research wave (quarterly minimum).

## Applies If (ALL must hold)

- After ≥6 user interviews, when patterns are emerging and the team needs shared shorthand.
- Onboarding new designers / PMs / engineers to a product's user base.
- Pre-feature discussions where "would the user want this?" keeps surfacing.
- B2B products with distinct buyer / champion / end-user roles that need separate treatment.

## Skip If (ANY kills it)

- Pre-research: "made-up" personas reinforce assumptions and push the team in the wrong direction.
- Pure technical infrastructure work where the user IS the developer.
- Single-customer enterprise projects — a stakeholder map is more useful than personas.
