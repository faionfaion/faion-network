# Performance Domains Overview

## Summary

PMBoK 7 defines eight interconnected performance domains — Stakeholder, Team, Development Approach, Planning, Project Work, Delivery, Measurement, Uncertainty — that together enable successful project delivery. Domains are ongoing concurrent concerns, not sequential phases. The overview serves as a router and assessment harness: triage which domains need active management, select the relevant detailed methodologies, and score each domain green/yellow/red at project gates.

## Why

Process-based PM can feel mechanical and disconnected from outcomes. Teams follow steps without understanding value. Performance domains shift focus from activities to results: what must be true for the project to succeed, not what process must be followed. The domain interaction matrix (poor Stakeholder → Team + Delivery suffer) provides leading indicators rather than lagging post-mortems.

## When To Use

- Onboarding a new project: triage which domains need active management on this engagement
- Periodic health checks: score each domain RAG and surface the weakest two for PM attention
- Tailoring an engagement: decide which methodologies to instantiate vs skip based on domain weight
- Routing a multi-methodology agent: classify input into 1-3 domains, dispatch to detailed methodologies
- Mapping PMBoK 6 process documentation onto PMBoK 7 outcome-based view

## When NOT To Use

- Single-task work where a checklist suffices — framework overhead exceeds value
- Pure Scrum teams for whom Scrum Guide already defines roles, ceremonies, and metrics — redundant overlay
- Operations / BAU work that is not a project (no defined start/end, no unique outcome)
- Low-stakes internal work where domain assessment cost exceeds project cost
