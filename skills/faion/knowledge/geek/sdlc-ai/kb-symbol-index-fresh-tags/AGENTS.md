# Symbol Index as Always-Fresh LSP/ctags Artifact

## Summary

Maintain a committed, always-fresh symbol index of the codebase — generated by an LSP-based tool (Serena, SourceGraph, or `ctags --languages=...` with the universal-ctags `tags-from-lsp` workflow) — and expose it to coding agents as the canonical "where is this defined / where is this used" lookup. The index is rebuilt by a `pre-commit` hook + a CI job that fails when the committed index drifts from the live AST. Agents are instructed (in the project `AGENTS.md`) to consult the symbol index for definition / reference / call-graph queries before falling back to text grep, cutting context cost by 5-20x on large repos.

## Why

Coding agents waste enormous context grepping for symbols across multi-million-line repos; LSP-grade indices answer the same query in O(1) lookups with structured spans. Anthropic's own "Claude Code best practices" guide and Cognition's Devin postmortems both call out symbol-index miss as the dominant context-bloat source. A committed, drift-checked index also lets agents work in offline / sandboxed environments where running an LSP server is impossible. Universal-ctags ships fast indexers for 40+ languages; modern tools like Serena and oxc-resolver give LSP-quality output without a running daemon, making the artifact reproducible in CI.

## When To Use

- Any monorepo over ~50k LOC where text grep blows the agent's context window.
- Polyglot repos where one tool (universal-ctags / Serena) covers every language uniformly.
- Sandboxed agent runtimes (Docker exec, GitHub Actions runners) where bringing up an LSP server is impractical.
- Refactor-heavy work where call-graph queries replace bulk file reads.

## When NOT To Use

- Single-file scripts or notebooks — index overhead dominates.
- Repos where the LSP server is already always-on in the agent runtime (Cursor desktop) and a local index just adds drift risk.
- Vendored / read-only dirs — index those once and exclude from drift checks.
