AUTOFIX-VS-FLAG POLICY (drop into the agent system prompt verbatim).

For every linter or scanner finding F you see:

AUTO-APPLY iff ALL hold:
  1. kind(F) is one of: format, import-sort, type-annotation, dead-code,
     unused-import.
  2. tool(F) exposes --fix / --write / --autofix AND the rule for F has
     a deterministic fix block (ruff, biome, eslint --fix, prettier,
     semgrep --autofix on rules carrying `fix:`).
  3. Tests pass after the fix (run the test command; require exit 0).
  4. Diff is < 50 lines OR every change has the same rule ID.

Otherwise FLAG:
  - SAST without fix block (CodeQL, Semgrep without `fix:`, Snyk Code).
  - Secrets (gitleaks, trufflehog) — ROTATE FIRST, then clean up.
  - CVEs without a patched version (Snyk, Dependabot, Trivy).
  - License conflicts (FOSSA, licensee) — ALWAYS escalate to human.
  - Cognitive-complexity hotspots (SonarQube) — propose refactor as
    draft PR, human approves.

NEVER:
  - Bypass with --no-verify.
  - Decide based on a confidence score.
  - Land an autofix in the same commit that introduces unrelated changes.
  - Auto-bump a dep to "fix" a CVE if no patched version exists.

WHEN flagging, post the finding as a PR review comment or a draft PR
suggestion. Cite the rule ID and the source URL.
