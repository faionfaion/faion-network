# Agent Integration — AI-Assisted Specification Writing

## When to use
- Starting a new feature, service, or system where requirements are known informally but not structured
- Producing spec.md and design.md documents in the SDD lifecycle before implementation begins
- When a human has described intent but edge cases, dependencies, and acceptance criteria are not yet explicit
- Reviewing an existing specification for completeness, internal conflicts, and missing acceptance criteria
- Converting informal requirements (meeting notes, chat logs, PRDs) into structured SDD-format specifications

## When NOT to use
- When domain knowledge is highly specialized (medical, legal, financial regulatory) and the agent lacks sufficient context — garbage-in-garbage-out on domain-specific requirements
- When specifications are already complete and formally approved — agent review adds no value and risks introducing changes to finalized docs
- For pure infrastructure or configuration tasks where the spec is trivially derived from the implementation
- When no human reviewer is available — AI-generated specs must always have a human approval gate before implementation begins

## Where it fails / limitations
- AI cannot infer business rules, political constraints, or organizational priorities that were never stated — these require explicit human input
- Acceptance criteria generated by AI tend to be technically correct but miss subtle UX edge cases and domain-specific business rules
- Context window limits constrain how much existing codebase context can be included alongside the spec generation prompt — large systems require decomposed, phase-by-phase spec generation
- Specs generated from vague intent (1-2 sentence descriptions) reliably contain hallucinated requirements that seem plausible but conflict with actual product intent
- AI cannot validate that requirements are mutually consistent without a dedicated consistency-check pass

## Agentic workflow
A Claude subagent running the SDD specification workflow receives human intent as input, asks structured clarifying questions, then generates spec.md using the standard template (functional requirements in FR-X format, edge cases, acceptance criteria in Given-When-Then, dependencies, out-of-scope items). A second review pass checks for internal conflicts, missing acceptance criteria, and ambiguous language. The output is staged for human approval before any implementation tasks are created. This matches the INTENT → SPEC phase of the SDD-AI workflow where AI drafts and human approves.

### Recommended subagents
- `faion-sdd-execution` — quality gates for spec review, pattern/mistake memory from prior specs
- `faion-brainstorm` — diverge-converge ideation for complex system specs with uncertain requirements
- `faion-feature-executor` — downstream consumer of approved specs; drives implementation tasks

### Prompt pattern
```xml
<task>Generate a specification for the following feature.</task>
<context>
  <product>{{product_description}}</product>
  <users>{{user_personas}}</users>
  <constraints>{{technical_and_business_constraints}}</constraints>
  <intent>{{feature_intent}}</intent>
</context>
<requirements>
  1. List functional requirements in FR-X format (FR-1, FR-2, ...)
  2. Identify and document edge cases
  3. Write acceptance criteria in Given-When-Then format for each FR
  4. List dependencies (internal systems, external APIs, data sources)
  5. List explicitly what is OUT OF SCOPE
  6. Flag any assumptions made due to missing information
</requirements>
<format>Use the standard spec.md template from SDD methodology.</format>
```

```xml
<task>Review this specification for quality issues.</task>
<spec>{{spec_content}}</spec>
<checks>
  - Internal conflicts between requirements
  - Acceptance criteria missing for any FR
  - Untestable or ambiguous acceptance criteria
  - Missing out-of-scope section
  - Requirements that depend on undefined external systems
</checks>
<output>JSON: { issues: [{fr: "FR-X", type: "conflict|missing|ambiguous", detail: "..."}] }</output>
```

Spec consistency check (runs markdownlint + checks for Given-When-Then coverage):
```bash
#!/bin/bash
# check-spec.sh — validate spec.md completeness
SPEC="${1:?Usage: $0 <spec.md>}"

markdownlint "$SPEC" --config .markdownlint.json 2>&1

# Count FR entries
FR_COUNT=$(grep -cP '^- \*\*FR-\d+\*\*' "$SPEC" || grep -c 'FR-[0-9]' "$SPEC" || true)
# Count Given-When-Then blocks
GWT_COUNT=$(grep -c 'Given ' "$SPEC" || true)

echo "Functional requirements: $FR_COUNT"
echo "Given-When-Then blocks: $GWT_COUNT"

if [ "$GWT_COUNT" -lt "$FR_COUNT" ]; then
  echo "WARNING: $((FR_COUNT - GWT_COUNT)) requirements may be missing acceptance criteria"
fi
```

## Best practices
- Always include an explicit "Out of scope" section — AI will attempt to fulfill implied requirements if scope boundaries are missing
- Break large system specs into 5-6 focused phases of 30-50 requirements each; this stays within model consistency limits
- Provide concrete examples in the input prompt ("like X but different in Y way") — AI generates more accurate requirements from analogies than from abstract descriptions
- After AI generates the spec, run a dedicated consistency-check pass with a separate agent call before human review — catch structural issues before the human reviews
- Version control every spec iteration in git — spec evolution is part of the audit trail and enables tracing requirement changes to code changes
- State non-goals explicitly ("this spec does not cover authentication, billing, or multi-tenancy") — LLMs will generate requirements for these if not excluded

## AI-agent gotchas
- Agents generating specs from brief descriptions often invent reasonable-sounding but incorrect business rules — human must review every FR, not just the structure
- Given-When-Then acceptance criteria generated by agents frequently test the implementation path rather than the requirement — humans must verify criteria are behavior-focused, not implementation-focused
- Agents with large context windows (200k) still lose coherence in very long specs — keep individual spec files under 15,000 tokens for consistent output quality
- If an agent generates a spec and then immediately proceeds to implementation planning, it may never surface the clarifying questions that would have caught ambiguity — separate the spec generation call from the plan generation call
- Agents will not proactively flag when they lack domain context; they generate plausible-sounding requirements instead — always prompt explicitly: "List any assumptions you made and any information you are missing"
