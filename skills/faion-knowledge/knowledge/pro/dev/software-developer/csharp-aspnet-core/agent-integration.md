# Agent Integration — ASP.NET Core Patterns

## When to use
- Building HTTP APIs in .NET 8/9 with `[ApiController]` controllers, DI-injected services, Entity Framework Core.
- Layered architecture (Controller → Service → Repository) with mapping via AutoMapper or manual mappers.
- Token-authenticated APIs using `[Authorize]` + JWT bearer or cookies.
- OpenAPI-first endpoints generated by Swashbuckle or NSwag, paired with strongly-typed DTOs.

## When NOT to use
- Microservices that fit minimal API style — controllers add ceremony; use `app.MapGroup("/api/users").MapGet(...)` instead.
- Reactive / streaming endpoints — switch to gRPC or SignalR.
- CQRS-heavy domains — service-per-controller pattern collapses; use MediatR + handlers per command/query.
- AOT-compiled apps — reflection-heavy patterns (AutoMapper, generic filters) break in Native AOT; prefer source generators.

## Where it fails / limitations
- "Repository over EF Core" is usually redundant — `DbContext` already implements UoW + Repository. Adding another layer hurts unless you have multiple data sources.
- AutoMapper magic strings break refactors silently; consider `Mapperly` (source-generated) or hand-written mappers.
- `IPasswordHasher<User>` requires Identity infra; agents inject it without realizing the cost.
- Controllers that return `ActionResult<T>` mix domain results with HTTP shapes — testing requires `Result.Value as OkObjectResult`.
- Service abstractions (`IUserService`) where there's only one impl add testing-only ceremony — many teams skip them now and mock `DbContext` via `EF.InMemory` or Testcontainers.

## Agentic workflow
A subagent generates the vertical slice in one diff: DTOs (records) → service interface + impl → controller → mapper profile → integration test using `WebApplicationFactory<Program>` + Testcontainers Postgres. Use `[ApiController]` + global exception handler (`IExceptionHandler` in .NET 8+) so controllers stay thin. Prefer `Results.Problem(...)` (RFC 7807) for error responses. For new projects, default to Minimal APIs — controllers only when you need filters/conventions.

### Recommended subagents
- `faion-sdd-executor-agent` — TDD cycle.
- An `aspnet-slice` subagent (project-local) — generates 6 files (DTOs, service, controller, mapper, EF migration, integration test).

### Prompt pattern
```
Add ApiKey vertical slice in MyApp.Features.ApiKeys:
- Records: CreateApiKeyDto(string Name), ApiKeyDto(int Id, string Name, DateTime CreatedAt)
- IApiKeyService { Task<ApiKeyDto> CreateAsync(CreateApiKeyDto, CancellationToken) }
- ApiKeyService uses AppDbContext, hashes secret via SHA256, returns DTO
- ApiKeysController POST/DELETE/GET with [Authorize], CreatedAtAction
- EF migration AddApiKeys
- Integration test using WebApplicationFactory + Testcontainers Postgres
Use record DTOs (not classes). Use TimeProvider.GetUtcNow(), not DateTime.UtcNow.
```

## Templates & scripts
See `templates.md` for full slice. Snippet — global error handler (.NET 8+):

```csharp
// ProblemDetailsHandler.cs
public class ProblemDetailsHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext ctx, Exception ex, CancellationToken ct)
    {
        var pd = ex switch
        {
            NotFoundException nf => new ProblemDetails { Status = 404, Title = nf.Message },
            ValidationException ve => new ProblemDetails { Status = 422, Title = "Validation failed", Detail = ve.Message },
            _ => new ProblemDetails { Status = 500, Title = "Server error" }
        };
        ctx.Response.StatusCode = pd.Status!.Value;
        await ctx.Response.WriteAsJsonAsync(pd, ct);
        return true;
    }
}

// Program.cs
builder.Services.AddProblemDetails();
builder.Services.AddExceptionHandler<ProblemDetailsHandler>();
app.UseExceptionHandler();
```

## Best practices
- Use record DTOs with required init properties — immutability for free.
- Return `Results<Ok<T>, NotFound, ValidationProblem>` (typed results) in Minimal APIs for OpenAPI accuracy.
- Inject `TimeProvider` instead of `DateTime.UtcNow` — testable + replaces `IClock` patterns.
- Centralize error mapping in `IExceptionHandler` + `AddProblemDetails()`.
- Prefer `services.AddDbContext<T>(o => o.UseNpgsql(...))` with explicit connection strings from `IConfiguration`; avoid hard-coded names.
- Always pass `CancellationToken` to async EF calls (`ToListAsync(ct)`, `SaveChangesAsync(ct)`); without it, a client disconnect leaves queries running.
- For pagination, use keyset (`WHERE id > @after ORDER BY id LIMIT 20`), not `Skip/Take` on large tables.
- Wire OpenTelemetry for traces + metrics; one block in `Program.cs` covers ASP.NET, EF, HttpClient.

## AI-agent gotchas
- Agents inject `DbContext` directly into singleton services — captive dependency, broken concurrency. Verify scope.
- They forget `await using var transaction = await db.Database.BeginTransactionAsync(ct);` for multi-step writes.
- LLMs add `Result<T>` wrappers around every method, then have controllers do `if (result.IsSuccess) return Ok(...)` — choose either typed results OR exception-based, not both.
- Generated tests use `MockBehavior.Default` and don't assert call counts; require `MockBehavior.Strict` or shift to integration tests.
- Agents return EF tracked entities directly to clients, exposing internal columns/circular refs. Mandate DTO mapping at the service boundary.
- Human-in-loop checkpoint: review middleware order in `Program.cs` — `UseAuthentication` must come before `UseAuthorization`, etc. Agents shuffle these and break auth silently.
