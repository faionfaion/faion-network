# Agent Integration — Personas

## When to use
- After ≥6 user interviews, when patterns are emerging and the team needs shared shorthand.
- Onboarding new designers / PMs / engineers to a product's user base.
- Pre-feature discussions where "would the user want this?" keeps surfacing.
- B2B products with distinct buyer / champion / end-user roles that need separate treatment.

## When NOT to use
- Pre-research: "made-up" personas reinforce assumptions and push the team in the wrong direction.
- Pure technical infrastructure work where the user IS the developer.
- Single-customer enterprise projects — a stakeholder map is more useful than personas.

## Where it fails / limitations
- Personas decay rapidly — products and audiences shift faster than personas are updated.
- Demographic-heavy personas (age, gender) tell you nothing actionable.
- Teams generate too many (8+) and remember none.
- Personas without scenarios are wallpaper — pretty, ignored.
- LLM-generated personas inherit dataset biases; they look real but reproduce stereotypes.

## Agentic workflow
Use a Claude subagent to cluster interview transcripts and survey data into candidate persona patterns. The agent surfaces clusters with evidence quotes; humans name and shape personas, never the other way around. Persona docs live next to the research corpus so re-clustering is cheap. Re-run after every meaningful research wave (quarterly minimum).

### Recommended subagents
- `faion-ux-researcher-agent` — clusters transcripts, surfaces candidate personas with quote evidence.
- `faion-usability-agent` — validates personas against fresh usability findings.
- General Claude subagent — drafts proto-personas from sales/support tickets when interviews are unavailable.

### Prompt pattern
```
Read these <N> interview transcripts. Group participants into
2–5 clusters by goals + behaviors (NOT demographics). For each
cluster output: 3 representative verbatim quotes, 3 shared
goals, 3 shared frustrations, 1 contrasting cluster. Reject
clusters supported by fewer than 2 participants.
```

```
Given persona <name> and feature <X>, write a 1-paragraph
scenario showing them succeeding with <X>, then a parallel
scenario showing them failing. Use only behaviors and contexts
already documented in the persona file.
```

## Templates & scripts
See `templates.md` for the full Persona and Proto-Persona templates. Inline cluster summarizer:

```python
#!/usr/bin/env python3
# cluster-summary.py — group rows by cluster column and emit goal/pain summaries.
import pandas as pd, sys
df = pd.read_csv(sys.argv[1])  # cols: participant_id, cluster, goal, pain
for c, g in df.groupby("cluster"):
    print(f"## Cluster {c} (n={g.participant_id.nunique()})")
    print("- Top goals:", g.goal.value_counts().head(3).to_dict())
    print("- Top pains:", g.pain.value_counts().head(3).to_dict())
    print()
```

## Best practices
- Cap personas at 3–5 primary, 1–2 secondary; anti-personas optional.
- Each persona must include at least one verbatim quote tied to a real participant.
- Always pair persona with a JTBD ("Sarah hires the dashboard to look prepared").
- Keep persona docs short (≤1 page); long personas get skimmed and forgotten.
- Date-stamp personas; mark as DRAFT until validated against ≥10 users.
- Track which decisions referenced which persona to measure usage.

## AI-agent gotchas
- LLMs synthesize plausible-sounding stereotypes; require quote-with-participant-ID for each trait.
- Photos generated by AI introduce demographic bias and accessibility issues — use illustrations or initials.
- Don't let the agent invent demographic attributes that weren't in source data.
- "Quote" fields are often paraphrased silently — instruct the agent to mark non-verbatim quotes.
- Agents will produce 8+ personas if not capped — enforce max-cluster constraint.
- Re-clustering on a different week will produce different personas; lock the corpus snapshot used.
