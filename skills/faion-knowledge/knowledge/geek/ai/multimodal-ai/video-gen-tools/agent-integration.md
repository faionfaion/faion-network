# Agent Integration — Video Generation Tools

## When to use
- Integrating Runway, Luma, or Replicate into an automated content pipeline
- Building a multi-provider video generation service with fallback logic
- Extending an existing video by appending new AI-generated segments
- Downloading and storing generated clips locally for downstream processing
- Wrapping video generation in async retry/timeout logic for production reliability

## When NOT to use
- Single one-off clip generation — use the web UI directly; API setup overhead is not justified
- Synchronous request/response architectures — all video APIs are async polling; wrong pattern will block
- When video quality is untested — validate provider outputs manually before automating at scale
- Embedding generation inside latency-sensitive user flows (<2s response expected)

## Where it fails / limitations
- `_upload_image` in `VideoGenerationService` is a stub — must be implemented per storage backend (S3, Cloudinary, etc.) before img2video works
- `RunwayVideoGenerator.generate_from_text` uses the image-to-video endpoint with no image, which is non-standard; verify endpoint with current SDK docs before use
- Replicate model hashes in code are pinned and can go stale as providers update models
- `VideoGenerationService.extend_video` uses frame extraction then img2video — seams are visible if the source clip was generated by a different provider
- Luma `_wait_for_completion` has no timeout; an infinite loop is possible if the API returns neither "completed" nor "failed"
- httpx async download in `_download_video` loads entire video into memory before writing — breaks on large files

## Agentic workflow
An agent receives a generation request with provider preference and prompt. It instantiates `VideoGenerationService`, calls `generate()`, awaits the async result, verifies the local path exists, then returns structured output with `{status, local_path, video_url}`. For multi-provider fallback, agents iterate `providers` list in order and stop on first success. A separate archival agent periodically sweeps `output_dir` and uploads to permanent storage (S3), since pre-signed video URLs expire.

### Recommended subagents
- `haiku` — Execute generation calls, poll, download, return structured result
- `sonnet` — Select provider based on prompt type (photorealistic → Luma; anime → Replicate/AnimateDiff; commercial → Runway)
- `haiku` — Video URL archival: sweep output dir, upload to S3, update metadata store

### Prompt pattern
```xml
<task>
Generate a video clip.
Provider: {{PROVIDER}}
Prompt: {{PROMPT}}
Duration: {{DURATION}}s
Aspect ratio: {{ASPECT_RATIO}}
Output dir: {{OUTPUT_DIR}}

Return JSON:
{"status": "success"|"failed", "local_path": "...", "error": "..."}
</task>
```

Fix for Luma infinite loop (add timeout):
```python
def _wait_for_completion(self, generation_id: str, timeout: int = 300) -> dict:
    import time
    start = time.time()
    while time.time() - start < timeout:
        response = requests.get(
            f"{self.base_url}/generations/{generation_id}",
            headers={"Authorization": f"Bearer {self.api_key}"}
        )
        gen = response.json()
        if gen["state"] == "completed":
            return {"status": "success", "video_url": gen["assets"]["video"], "id": generation_id}
        elif gen["state"] == "failed":
            return {"status": "failed", "error": gen.get("failure_reason"), "id": generation_id}
        time.sleep(5)
    return {"status": "timeout", "id": generation_id}
```

## Best practices
- Always download videos immediately after SUCCEEDED — pre-signed URLs expire in 30–60 minutes depending on provider
- Pin Replicate model hashes to a known-good version; record them in a config file, not inline in code
- Implement provider-level rate limiting (Runway: 10 concurrent tasks; Luma: varies by plan)
- Use streaming download for large files (`httpx` stream mode or `requests` with `stream=True`) to avoid OOM
- Log both the submitted prompt and the task ID together; re-querying failed tasks requires the task ID
- Store provider, model version, prompt, seed, and output path in a generation log for reproducibility
- Validate output file exists and size > 0 before returning success; API can return SUCCEEDED with empty output
- Use separate output directories per provider to simplify debugging and cost attribution

## AI-agent gotchas
- Runway SDK `image_to_video.create` requires an image URL even for text-only generation on some model versions; have a fallback blank image URL ready
- Replicate returns an iterator or a URL depending on model and output type; `output[0]` fails if output is not a list
- Luma `keyframes.frame0.type = "generation"` for extension requires the source generation ID, not the video URL — agents must track IDs, not just URLs
- `VideoGenerationService.extend_video` extracts frames from a local file then re-uploads; if the video has no audio, the extended version will also be silent
- Async `_upload_image` is a stub — any agent that calls `generate_from_image(image_path=...)` will hit `NotImplementedError` until implemented
- Task IDs and generation IDs are provider-specific and non-portable; never pass a Runway task ID to a Luma polling call
- Error messages from Runway on content policy violations are terse ("FAILED") with no explanation — agent must not retry without modifying the prompt
