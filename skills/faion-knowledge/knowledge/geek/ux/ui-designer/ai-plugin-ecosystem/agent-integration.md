# Agent Integration — AI Plugin Ecosystem

## When to use
- Automating repetitive Figma operations (renaming layers, bulk style changes, exporting assets) via Automator or similar
- Running accessibility audits across a design file before handoff using Stark or Able
- Generating placeholder content (text, images) at scale with Content Reel for prototype realism
- Selecting and acting on similar layers (Similayer) as part of a scripted design cleanup workflow
- Evaluating which plugin combination covers a team's workflow gaps before investing in custom UXP development

## When NOT to use
- When plugin output must be reproducible without human intervention — most AI plugins are not deterministic
- When the team lacks a Figma Organization or Professional plan (some plugins require paid tiers)
- When design decisions require brand or strategic judgment — plugins are automation tools, not decision-makers
- When a plugin relies on a third-party AI API that could expose proprietary design data to external servers
- When the design system is unstable — plugins that reference tokens/styles break when foundations change

## Where it fails / limitations
- Figma REST API does not expose a plugin execution endpoint; agents cannot trigger plugin runs programmatically from outside Figma
- AI content generated by plugins (Magician, etc.) is non-deterministic — same prompt yields different results across runs
- Most Figma AI plugins have no CLI or webhook interface; they only operate within the Figma desktop/browser UI
- Plugin data access is sandboxed — they can read/write the current file but cannot batch-process multiple files
- Accessibility plugins (Stark) flag issues but cannot auto-fix them; a human must apply remediations
- Plugin API rate limits and quota resets are opaque; heavily automated usage risks account suspension

## Agentic workflow
Agents cannot directly invoke Figma plugins — the plugin ecosystem operates inside the Figma canvas and requires a human-initiated session. The practical agentic pattern is to use the Figma REST API to read/write file data (nodes, styles, components), then post-process results externally with Claude. For accessibility, an agent can extract color contrast pairs via the API and run WCAG checks in code without needing the Stark plugin. For content generation (placeholders, copy), an agent produces content externally and pushes it via the Figma API's `PATCH /v1/files/{key}/nodes` endpoint. Reserve actual plugin use for human-driven workflows and use the REST API as the agent integration surface.

### Recommended subagents
- `faion-sdd-executor-agent` — executes Figma API calls as part of a design-system audit task
- Custom accessibility-check agent — reads Figma fills/strokes via REST, computes WCAG contrast ratios, returns a violation report

### Prompt pattern
```
You have access to a Figma file JSON export. Audit all text layers for:
1. Contrast ratio < 4.5:1 (normal text) or < 3:1 (large text)
2. Missing semantic role (decorative vs. informational not flagged)

Return a JSON array: [{node_id, name, issue, current_ratio, required_ratio}]
```

```python
import os, requests, math

TOKEN = os.environ["FIGMA_TOKEN"]
FILE_KEY = os.environ["FIGMA_FILE_KEY"]

def luminance(r, g, b):
    def c(v):
        v /= 255
        return v / 12.92 if v <= 0.04045 else ((v + 0.055) / 1.055) ** 2.4
    return 0.2126 * c(r) + 0.7152 * c(g) + 0.0722 * c(b)

def contrast(c1, c2):
    l1, l2 = luminance(*c1), luminance(*c2)
    if l1 < l2:
        l1, l2 = l2, l1
    return (l1 + 0.05) / (l2 + 0.05)

def hex_to_rgb(h):
```

## Best practices
- Treat Figma plugins as human-operated tools and the Figma REST API as the agent integration boundary
- Maintain a curated plugin list per project in AGENTS.md to prevent team members from installing unreviewed plugins that could leak design IP
- For accessibility, run both a plugin (Stark, during design) and an automated axe-core scan (post-handoff, on rendered HTML) — neither alone is sufficient
- When evaluating a new AI plugin, test on a disposable copy of the file, not the main design system
- Log which plugin version produced which design output for auditability — AI plugin outputs change with model updates
- For batch operations that agents could handle, prefer the Figma REST API over plugin-based automation to maintain repeatability

## AI-agent gotchas
- There is no way for an agent to trigger a Figma plugin run; any workflow described as "agent runs plugin" requires a human at the keyboard
- Figma REST API write access is restricted: you cannot create new frames, only update existing node properties
- File JSON exports from Figma REST can be large (10–200 MB for complex files); agents must paginate or scope requests to specific node IDs
- Plugin sandbox isolates execution — plugins cannot call external APIs unless the plugin author has set up a proxy; proprietary prompts in plugin fields may be sent to third-party servers
- Figma token scopes (read vs. write) must be set correctly; a read-only token will silently fail write operations
- The Figma Community plugin listing does not indicate whether a plugin sends data externally; review plugin source code or privacy policy before use with proprietary designs
