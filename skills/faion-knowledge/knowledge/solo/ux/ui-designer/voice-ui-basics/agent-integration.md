# Agent Integration — Voice UI Design Basics

## When to use
- Designing a new voice feature for a mobile app, smart speaker skill, or IVR system
- Writing and reviewing dialogue scripts for conversational AI products
- Defining intent/slot schemas for NLU training (Alexa, Dialogflow, Rasa)
- Auditing existing voice flows for error handling gaps and confirmation strategy mismatches
- Generating multiple utterance variations for NLU training data

## When NOT to use
- Complex data entry, browsing, or detailed comparison tasks — voice is the wrong modality
- Noisy environments or contexts requiring private information (credit card numbers, passwords)
- First-version MVP features where basic screen UI is not yet validated — voice adds a second modality before the first is proven
- When the deployment platform (web, desktop app) has no microphone access or TTS output

## Where it fails / limitations
- Agents generate dialogue scripts that sound natural in text but are awkward when spoken aloud — always read prompts aloud before approving
- Intent/slot schemas generated without real user utterance data have poor NLU coverage; agents tend to write idealized utterances, not natural speech
- Error recovery chains generated by agents are often too short (2 attempts, then fallback) or too long (endless reprompts)
- Agents cannot simulate speech recognition errors, accent variation, or ambient noise — Wizard of Oz testing with real users is irreplaceable
- Multi-turn dialogue context management is routinely under-designed by agents working from single-turn examples
- Platform-specific constraints (Alexa response length, Google Assistant card limits) are often ignored in generated flows

## Agentic workflow
An agent receives a use-case description and generates: (1) sample dialogues for happy path and top 3 error paths, (2) intent/slot schema in the target platform format (Alexa JSON, Dialogflow YAML, or Rasa NLU YAML), and (3) a set of 10-15 utterance variations per intent for NLU training. A second agent reviews for prompt length violations, confirmation strategy gaps, and missing fallback routes. Human Wizard of Oz testing validates before NLU training begins.

### Recommended subagents
- General Claude subagent (opus) — dialogue script writing and intent schema design require nuanced language modeling
- General Claude subagent (sonnet) — utterance variation generation for NLU training data

### Prompt pattern
```
You are a VUI designer. Given this use case: [description]
Write:
1. Happy path dialogue (user + system turns, 4-6 turns max)
2. Error path #1: speech not recognized (3-attempt recovery chain)
3. Error path #2: slot value invalid (with examples offered to user)
4. Intent schema with required/optional slots and 10 utterance variations
Follow these constraints: each system prompt ≤20 words when spoken aloud; no "I'm sorry"
as a standalone response; always offer an alternative path on 3rd error.
```

```
Utterance variation generator:
Given this intent: [IntentName] with slots [slot1, slot2]
Generate 20 natural utterance variations that a real user might say.
Include: different word orders, contractions, implicit slot omissions, informal speech.
Flag any variation that might conflict with [list of other intents].
```

Inline script — validate dialogue script for prompt length and common anti-patterns:
```python
#!/usr/bin/env python3
"""
vui-lint.py — lint a VUI dialogue script for common issues
Usage: python vui-lint.py dialogue.txt
Format: lines starting with "System:" are system prompts
"""
import sys, re

MAX_WORDS = 25  # spoken prompts should be short
ANTI_PATTERNS = [
    (r"I'm sorry", "Avoid hollow apology without action"),
    (r"Please (hold|wait)", "Avoid dead air without context"),
    (r"\d{4,}", "Long numbers are hard to hear — spell out or chunk"),
    (r"\.com|\.net|\.org", "URLs in voice prompts are unusable"),
]

with open(sys.argv[1]) as f:
    lines = f.readlines()

issues = []
for i, line in enumerate(lines, 1):
    if not line.startswith("System:"):
        continue
    text = line[7:].strip()
    words = len(text.split())
    if words > MAX_WORDS:
        issues.append(f"Line {i}: too long ({words} words > {MAX_WORDS}): {text[:60]}...")
    for pattern, msg in ANTI_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            issues.append(f"Line {i}: {msg}: {text[:60]}...")

if issues:
    print("\n".join(issues))
    sys.exit(1)
else:
    print("No issues found.")
```

## Best practices
- Write dialogues in spoken form first, then extract intents — never design intents before writing dialogues
- Keep system prompts under 20 words for smart speakers; under 15 for IVR; under 30 for screen+voice hybrids
- Every intent needs at minimum 10 utterance variations before NLU training; 20+ is the practical minimum for production
- Use implicit confirmation for low-risk actions, explicit for irreversible ones — never confirm everything (users abandon verbose confirmation)
- Design the third-error fallback before designing the happy path — fallbacks are where voice UIs fail most visibly
- Test with at least 3 different accents and 2 ambient noise levels before shipping

## AI-agent gotchas
- Agents read dialogue scripts visually; they do not simulate speech — read every system prompt aloud before approving
- Intent schemas generated without real utterance data will underperform in production; Wizard of Oz testing is a mandatory human checkpoint
- Agents generating NLU training data tend to produce syntactically varied but semantically homogeneous utterances — add real user quotes from research
- Multi-turn context tracking (remembering slot values across turns) is routinely omitted in agent-generated flows; explicitly request context persistence handling
- Platform length limits are frequently violated: Alexa SSML responses have a 8000-character limit; Google Assistant display responses have card constraints
