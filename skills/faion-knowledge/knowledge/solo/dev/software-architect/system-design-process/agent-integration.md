# Agent Integration — System Design Process

## When to use
- Greenfield system design where requirements, NFRs, and scale assumptions are still being captured.
- Replacement/replatform decisions that need a documented spec, ADRs, and C4 diagrams before any code.
- Pre-implementation handoff: turning a product brief into a buildable architecture package the dev agent can execute.
- Internal interview-style design exercises where a reviewable artifact (not a whiteboard photo) is the output.

## When NOT to use
- Small bug fixes or localized refactors — overkill, use the relevant pattern methodology directly.
- Prototype/spike code where the goal is to learn, not to commit to an architecture.
- Domains the team already operates daily (CRUD admin tools) — skip to templates.
- Operational incidents — use reliability-architecture and observability instead.

## Where it fails / limitations
- LLM scale estimates anchor on examples in training data; numbers can be off by an order of magnitude. Always validate with real product metrics.
- The 5-phase flow assumes stable requirements; with shifting product direction, design churn explodes. Pause and re-spec instead of redesigning component-by-component.
- Pattern recommendations bias toward popular solutions (microservices, Kafka, K8s) even when a monolith would do. Force the agent to argue against its first suggestion.
- C4 diagrams generated by LLMs frequently invent components. Must be reviewed against actual repo structure.

## Agentic workflow
Drive this process as a sequential pipeline: spec agent → architecture diverge agent → trade-off converge agent → ADR/C4 author agent → reviewer. Use `faion-brainstorm` for the diverge/converge cycle on architecture style and key components, then hand the chosen design to `faion-feature-executor` for SDD task breakdown. Keep the human in the loop at two checkpoints: NFR sign-off and ADR acceptance — everything else can be agent-driven against `templates.md`.

### Recommended subagents
- `faion-brainstorm` — diverge/converge on architecture options and trade-offs.
- `faion-sdd-execution` — turn accepted design into spec.md/design.md/test-plan.md per SDD lifecycle.
- `faion-feature-executor` — execute the implementation plan once the design is locked.
- `faion-improver` — post-build audit against the original NFRs and ADRs.

### Prompt pattern
```
You are designing <system> for <users>. NFRs: <p95 latency, RPS, availability, RPO/RTO>.
1. List 3 architecture options with one trade-off each.
2. Pick one, justify against NFRs.
3. Output Mermaid C4 L1+L2 and one ADR per non-obvious decision.
Never invent components not justified by an NFR.
```

```
Stress-test the design above. Identify: single points of failure, hot partitions,
data-loss windows, fan-out amplifiers, and silent-failure modes. Be specific —
no "consider monitoring" filler.
```

## Templates & scripts
See `templates.md` for ADR + Mermaid C4 examples. Inline helper to bootstrap an architecture pack inside a repo:

```bash
#!/usr/bin/env bash
# bootstrap-arch-pack.sh — scaffold spec/ADR/C4 dirs an agent can fill.
set -euo pipefail
ROOT="${1:-docs/architecture}"
mkdir -p "$ROOT"/{decisions,diagrams,specs}
cat > "$ROOT/decisions/0000-record-architecture-decisions.md" <<'EOF'
# 0000. Record architecture decisions
Date: TBD
Status: accepted
Context: We need a record of decisions.
Decision: Use Markdown ADRs in docs/architecture/decisions.
Consequences: Each significant decision becomes a numbered ADR.
EOF
cat > "$ROOT/diagrams/context.mmd" <<'EOF'
C4Context
title System Context
Person(user, "User")
System(sys, "System", "Replace me")
Rel(user, sys, "Uses")
EOF
cat > "$ROOT/specs/spec.md" <<'EOF'
# Spec
## Functional Requirements
## Non-Functional Requirements
| Attr | Target | Source |
|------|--------|--------|
EOF
echo "scaffold ready in $ROOT"
```

## Best practices
- Force NFR numerics before any diagram: an LLM with vague NFRs will always over-engineer.
- Make the agent produce three options for every non-trivial choice and explicitly score them against NFRs — single-option output is the failure signature.
- Keep ADRs in-repo as Markdown; route every PR that changes architecture-relevant code through an ADR review check.
- Pin the agent to actual repo structure (paste `tree -L 2`) before asking for component diagrams — eliminates fabricated services.
- Run `mmdc` in CI to fail builds when Mermaid in `docs/` no longer parses; treat diagrams as code.
- Do back-of-envelope math in the prompt; do not let the agent skip to architecture without writing RPS/storage/bandwidth.

## AI-agent gotchas
- Agents will often skip the "validate" phase. Make it an explicit subagent call with a different prompt, not a closing paragraph.
- Long Q&A "understand" phases blow context windows; checkpoint into `spec.md` and have the next agent re-read from disk.
- Mermaid C4 syntax has version drift — agents mix `C4Context` keywords with classic `flowchart`. Lint with `mmdc -i`.
- Trade-off matrices generated by LLMs frequently use the same scores for every option ("8/10" everywhere). Require differentiation in the prompt.
- Human-in-loop checkpoints: NFR sign-off (before design), ADR review (before implementation), pre-launch validation (against initial NFRs). Agents should halt and request approval at these gates.
