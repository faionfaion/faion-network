# Agent Integration — Serverless Architecture

## When to use
- Designing event-driven backends (HTTP APIs, queue processors, scheduled jobs) that need pay-per-execution and auto-scale-to-zero economics.
- Generating IaC (SAM, CDK, SST, Serverless Framework, Terraform) for Lambda/Functions/Workers from a feature spec.
- Optimizing existing Lambda functions: right-sizing memory, choosing ARM/Graviton, eliminating cold starts, reviewing IAM least-privilege.
- Composing multi-step workflows with Step Functions, EventBridge, SQS/SNS where orchestration logic is the deliverable.
- Ad-hoc cost-modeling: `Lambda$ vs Fargate$` crossover, GB-seconds estimation, Provisioned Concurrency ROI.

## When NOT to use
- Long-running tasks (>15 min Lambda hard limit), heavy compute (video transcoding, GPU/ML training), or persistent WebSocket fan-out — push the agent toward Fargate, ECS, GKE Autopilot, or AppSync/API Gateway WS instead.
- Latency-critical paths needing <50ms p95 with cold-start variance unacceptable (use containers + keep-warm or provisioned concurrency, but reconsider serverless fit first).
- Workloads with steady ~10M+ req/month at consistent load — agent should run cost comparison and recommend containers.
- Compliance regimes requiring physical isolation, custom kernels, or air-gapped deploys.

## Where it fails / limitations
- Agents hallucinate AWS quotas, runtime versions, region availability, and pricing — these change quarterly. Always verify with `aws lambda get-account-settings` or pricing API.
- Generated IAM policies often grant `*` on resources or actions; agents must be forced to scope with `Resource: !GetAtt` and `Condition` blocks.
- Cold-start estimates are runtime/memory/dependency dependent — LLM cannot benchmark, only suggest.
- Cross-account, multi-region, and VPC-attached Lambda configs need human review for ENI exhaustion, NAT GW costs, and KMS grant propagation.
- Step Functions ASL JSON generated by LLMs frequently misuses `ResultPath`/`OutputPath`, mismatches `States.TaskFailed` retriers, and skips `IntrinsicFunctions` quoting.

## Agentic workflow
Treat serverless design as a 3-phase pipeline: design (sonnet for trade-offs) → IaC scaffold (haiku from templates.md) → review (opus for IAM, cold-start, cost). Drive a single subagent per Lambda or workflow with the methodology folder as context, output IaC + handler skeleton + observability instrumentation, then route the diff to a reviewer subagent before commit. Never let one agent both author and approve IAM.

### Recommended subagents
- `faion-sdd-executor-agent` — owns end-to-end SDD task (spec → SAM/CDK + handler + tests), reads `templates.md` for boilerplate.
- `nero-sdd-executor-agent` — same role, scoped to NERO repos with NERO secret/credential conventions.
- `password-scrubber-agent` — sweeps generated env vars and CFN outputs for hardcoded secrets before commit; mandatory on serverless commits because Lambda env vars are a common leak vector.

### Prompt pattern
```
Design a Lambda + EventBridge pipeline for <event source> → <transform> → <sink>.
Constraints: runtime=python3.13, memory<=512MB, p95<2s, region=eu-central-1, no VPC.
Output: SAM template, handler with init-outside-handler pattern, OTEL instrumentation,
IAM scoped to specific resource ARNs only. Flag anything you assume.
```

```
Audit this Lambda for cold start: <paste handler + template>.
Report: import-time work, sync init blocking calls, package size, recommended
memory/arch change with $/month delta, and SnapStart/Provisioned Concurrency fit.
```

## Templates & scripts
See `templates.md` for SAM, CDK, SST, Serverless Framework starters. Inline cost-tuning helper for an agent loop:

```bash
#!/usr/bin/env bash
# power-tune.sh — drive aws-lambda-power-tuning Step Function for one function
set -euo pipefail
FN_ARN="$1"
PAYLOAD_FILE="${2:-payload.json}"
SM_ARN="${POWER_TUNING_SM_ARN:?must be set}"

aws stepfunctions start-execution \
  --state-machine-arn "$SM_ARN" \
  --input "$(jq -n --arg arn "$FN_ARN" --slurpfile p "$PAYLOAD_FILE" '{
      lambdaARN: $arn,
      powerValues: [128, 256, 512, 1024, 1769, 3008],
      num: 50,
      payload: $p[0],
      strategy: "balanced"
  }')" \
  --query executionArn --output text
```

## Best practices
- Group functions by bounded context (orders, payments) not by HTTP verb; one repo, one CFN stack, multiple functions sharing a layer.
- Init clients (DynamoDB, HTTP, secrets cache) at module scope so warm invocations skip TLS handshakes.
- Make every handler idempotent with a deterministic key in DynamoDB (TTL'd) — Lambda retries are at-least-once across SQS, EventBridge, and async invokes.
- Default to ARM/Graviton (`Architectures: [arm64]`) — 20% cheaper, often faster, no code change for Python/Node/Go.
- Wrap external HTTP calls with `botocore.config.Config(retries={'mode': 'adaptive'})` and bound timeouts; never inherit the function timeout as the HTTP timeout.
- Ship logs as structured JSON (Powertools for AWS Lambda) so agent-driven log search via `aws logs start-query` actually works.
- Scope IAM to resource ARNs and add `aws:SourceArn` / `aws:SourceAccount` conditions on cross-service trust.

## AI-agent gotchas
- LLMs default to `arn:aws:s3:::*` and `dynamodb:*` — explicitly demand `Resource:` lists in the prompt and reject diffs that contain `*` in `Resource`.
- Cold-start timing in answers is anecdotal; route any latency claim to `aws-lambda-power-tuning` rather than trusting LLM numbers.
- Generated `Layers:` ARNs are often wrong (region/version drift). Always parameterize via SSM Public Parameters or fetch the latest ADOT layer in CI.
- Agents skip the `DLQ` / `OnFailure` destination on async functions — checkpoint a human review when no DLQ is present.
- Step Functions ASL needs human-in-loop for any state with `Catch` + compensating logic; one wrong `ResultPath` can swallow errors silently.
- Any Lambda touching production data should be gated behind `terraform plan` / `cdk diff` review — never let an agent apply IAM or VPC changes unattended.
- For Cloudflare Workers, agents often miss the 50 subrequest / 30s CPU limits and produce code that works locally but throws `Worker exceeded CPU time` in prod.
