# Agent Integration — Specification Template

## When to use
- Starting any new feature that will be executed by a subagent — the template enforces the FR/AC structure the executor needs
- When a stakeholder or user provides fuzzy requirements and you need a structured artifact to validate understanding
- Before writing `design.md`: spec must exist and be approved first
- Generating spec drafts from user interviews, chat logs, or product briefs

## When NOT to use
- Hot-fixes and patches — spec overhead exceeds value for changes under ~2h
- Pure infrastructure tasks with no user-facing behavior
- Research spikes where requirements are deliberately undefined

## Where it fails / limitations
- The template's "User Personas" section adds little value in a solo solopreneur context — skip or collapse to a single persona
- "Estimate" fields on User Stories conflict with the project rule of no time estimates — use token estimates or T-shirt size only
- Template does not enforce MoSCoW on NFRs — agents filling the template often leave NFR priority blank

## Agentic workflow
A spec-writing agent receives a product brief or feature description and fills the template section by section: first the Problem Statement and User Stories, then Functional Requirements (one FR per story), then ACs in Given-When-Then. The agent must not invent technical decisions (those go in design.md). A review pass checks that every FR-X traces to a US and every AC traces to a FR.

Running spec generation and spec review as separate agent calls catches omissions that a single-pass agent misses.

### Recommended subagents
- `faion-sdd-executor-agent` — can generate a spec stub from implementation-plan context when working backwards
- General Claude subagent (Opus) — best for ambiguous problem-to-spec translation requiring judgment

### Prompt pattern
```
Using the spec template, write a spec.md for this feature: <description>.
Sections required: Problem Statement, User Stories (max 5), Functional Requirements,
Acceptance Criteria (Given-When-Then), Out of Scope.
Do NOT include implementation details — those go in design.md.
Label every FR with MoSCoW priority.
```

```
Review this spec.md for completeness. Check:
1. Every US has at least one FR.
2. Every FR has at least one AC.
3. Every AC is testable (no vague language).
4. Out of Scope is explicit.
Report missing coverage as a table.
```

## Templates & scripts
See `template-spec/README.md` for the full copy-paste template (this methodology IS the template). Also see `spec-structure/README.md` for v2.0 extended template with Personas and Appendix.

Validation script — check that every FR in spec traces to a US:
```bash
#!/usr/bin/env bash
# spec-trace-check.sh SPEC_FILE
SPEC=${1:?usage: spec-trace-check.sh spec.md}
FRS=$(grep -oP 'FR-\d+' "$SPEC" | sort -u)
for fr in $FRS; do
  if ! grep -q "$fr" "$SPEC"; then
    echo "MISSING TRACE: $fr has no US or AC reference"
  fi
done
echo "Trace check complete for $SPEC"
```

## Best practices
- Write the Problem Statement before anything else — if you cannot fill "Who / Problem / Impact / Solution / Metric", the feature is not ready to spec
- Keep FR text short and unambiguous: "System SHALL allow X" — one verb, one object
- Explicitly list at least three Out of Scope items; this prevents scope creep during design phase
- Status field (`Draft → Review → Approved`) is a human gate — never hand a Draft spec to an executor agent
- Condense the template for MVP features: Problem Statement + User Stories + FRs + ACs + Out of Scope is sufficient (see `spec-examples-basic/README.md`)

## AI-agent gotchas
- Agents tend to merge design decisions into specs — any sentence describing HOW something works belongs in design.md, not spec.md; enforce this with a review prompt
- "Acceptance Criteria" sections generated by agents often contain non-testable criteria ("user experience is smooth") — run a dedicated AC-quality check pass
- Template version drift: if spec.md and design.md use different FR numbering schemes, executor agents get confused — validate numbering consistency before task creation
- Human approval of spec.md is a required checkpoint before design begins — agents must not auto-proceed
