# Agent Integration — Backlog Grooming & Roadmapping

## When to use
- Weekly or monthly: agent-assisted grooming session to score, reorder, and refine the backlog in `.aidocs/backlog/`
- When the backlog exceeds 20 items and prioritization has not been revisited in over 2 weeks
- When starting a new quarter: generate or update roadmap.md with Now/Next/Later horizons from the current backlog state
- When a human provides a rough feature list and wants structured spec stubs with RICE scores
- When performing MVP scoping: run MoSCoW analysis against a candidate feature set before committing to implementation

## When NOT to use
- As a replacement for human strategic decisions — prioritization of P0 items requires business context the agent cannot access
- During active feature execution — backlog grooming is planning work; mix it with execution work and both suffer
- When the roadmap.md does not yet exist — create it first with a human-defined horizon and then let agents refine it
- For grooming items that require stakeholder input the agent cannot simulate (pricing decisions, partnership dependencies)

## Where it fails / limitations
- RICE scoring by an agent is always estimated; Reach and Impact scores require real user data the agent does not have — treat agent-generated RICE scores as hypotheses, not decisions
- Agents produce overly polished backlog item descriptions that mask uncertainty; text quality ≠ idea quality — require confidence annotations
- LLMs cannot distinguish strategic P0s from tactical P0s without explicit business context; they tend to assign high scores to technically interesting items
- Roadmap.md dates generated by agents are fictional; use Now/Next/Later format to avoid false precision
- Agents do not prune backlogs autonomously — they add items but rarely archive or drop them; backlogs grow unbounded without explicit cleanup instructions

## Agentic workflow
An agent runs a grooming session by reading all items in `.aidocs/backlog/`, scoring each with RICE (using estimates flagged as hypotheses), ranking the list, and writing updated spec stubs for the top-5 items. For roadmap updates, the agent reads the current roadmap.md, reads the constitution.md for strategic constraints, and proposes changes to the Now/Next/Later horizons based on current backlog scores and completed items in done/. All agent output is marked as a draft requiring human review before items move to todo/.

### Recommended subagents
- `faion-sdd-executor-agent` — not appropriate for grooming (it executes, not plans); use a planning-mode agent instead
- Sonnet-tier subagent — RICE scoring, gap analysis, acceptance criteria generation from rough feature descriptions
- Opus-tier subagent — roadmap synthesis, strategic theme identification, OKR-to-feature mapping

### Prompt pattern
```
Read all feature folders in .aidocs/backlog/.
For each feature, read README.md or spec.md if present.
Score each using RICE (flag all scores as estimates, not data):
- Reach: 1-10 (how many users per quarter, rough estimate)
- Impact: 0.25/0.5/1/2/3 (per-user impact)
- Confidence: 50%/80%/100% (your certainty in these estimates)
- Effort: 0.5/1/2/3/6 (person-months to complete)
Output a ranked table: Feature | R | I | C | E | RICE Score | Est. Tokens
Then write a one-paragraph rationale for the top-3 items.
```

```
Read roadmap.md and .aidocs/done/ (completed features).
Propose updates to roadmap.md:
1. Move completed features out of Now horizon
2. Promote top-scored backlog items into Next horizon
3. Flag any Now-horizon items that have been stalled for 2+ sprints
4. Remove items from Later that are >12 months old with no progress
Output the proposed diff as a patch, not the full updated file.
Human will review and apply.
```

## Templates & scripts
See `templates.md` for backlog item YAML template, RICE scoring table, and Now/Next/Later roadmap skeleton.

RICE batch scorer (outputs scored CSV from backlog folder):
```bash
#!/usr/bin/env bash
# Usage: ./rice-score-backlog.sh
# Reads all .aidocs/backlog/*/README.md and prints RICE table header
# (Agent fills in actual scores via prompt; this just discovers items)
set -euo pipefail
echo "Feature | Reach | Impact | Confidence | Effort | RICE | Notes"
echo "--------|-------|--------|------------|--------|------|------"
for dir in .aidocs/backlog/*/; do
  name=$(basename "$dir")
  if [[ -f "$dir/README.md" ]]; then
    title=$(head -1 "$dir/README.md" | sed 's/^# //')
  else
    title="$name"
  fi
  echo "$title | ? | ? | ? | ? | ? | [estimate]"
done
```

MoSCoW classifier prompt scaffold:
```
Given this feature list for MVP release:
[list features]

Classify each as Must/Should/Could/Won't for MVP.
Constraints: Must items ≤ 60% of total scope. Won't items explicitly documented.
For each Won't: provide a one-sentence reason it is excluded from this release.
Output as a table: Feature | Category | Rationale
```

## Best practices
- Never mark more than 3 items as P0/Must in any single release — if everything is critical, nothing is; require the agent to enforce this constraint explicitly
- Always annotate agent-generated RICE scores with `[estimate]` — this signals to future readers (human and agent) that the score has not been validated with data
- Use Now/Next/Later over quarterly dates; the roadmap.md should communicate direction, not schedule; agents generating date-based roadmaps introduce false precision
- Separate grooming from status updates — a grooming session reads the backlog and produces a prioritized list; it does not ask about sprint progress
- Maintain a "Won't Do (This Release)" section in roadmap.md; explicit exclusions prevent the same items from being proposed in every grooming session
- Archive backlog items older than 6 months with no movement to `.aidocs/backlog/.archive/` — they clutter the active backlog and confuse RICE scoring

## AI-agent gotchas
- Agents scoring RICE assign high Confidence (100%) to almost everything because they lack data to justify lower confidence; default to 50-80% for agent-generated scores and require explicit justification for 100%
- The "AI confidence trap" (Humanizing Work) applies here: polished backlog items generated by an agent look certain and well-reasoned but may be pure hypothesis; always mark AI-generated items as hypothesis and require user validation before spec-writing
- Agents asked to groom a backlog rarely drop items — they reword and reorder instead; explicitly instruct the agent to identify items for archival or deletion
- Roadmap synthesis by agents produces theme labels ("User Growth," "Platform Stability") that sound strategic but are vacuous without OKR connections; require the agent to link each roadmap theme to a specific metric or constitution goal
- Agents cannot know which backlog items have stakeholder buy-in; items with high RICE scores may be impossible to execute without external agreement the agent is unaware of — human must validate strategic feasibility before moving to todo/
