# Agent Integration — Writing Design Documents

## When to use
- After a spec.md is approved and before writing any implementation-plan.md
- When a feature touches 5+ files, changes a DB schema, modifies an API contract, or introduces cross-service dependencies
- When the team/solo dev needs an explicit record of alternatives considered (to prevent re-litigating decisions later)
- When onboarding a new agent into an existing codebase — design.md gives it the full architectural context in one read

## When NOT to use
- Bug fixes touching 1-2 files with no architectural impact
- Small refactors with no interface changes
- Features whose entire scope fits in a single task.md under ~5k tokens
- Greenfield experiments / spikes where the design is deliberately exploratory and will be thrown away

## Where it fails / limitations
- Design docs become stale quickly in fast-moving projects; an outdated design.md is worse than none (agents treat it as truth)
- AD-X decisions written without real alternatives analysis produce pseudo-documentation — agents reuse them uncritically
- Design docs authored entirely by LLMs without human review of trade-off tables produce biased decisions (LLMs favor the option they generated first)
- File structure sections listing non-existent files confuse executor agents; they create orphaned files or skip critical ones
- API contract sections without concrete request/response examples lead to incompatible implementations between agents working in parallel

## Agentic workflow
An agent receives the approved spec.md and constitution.md, reads existing code patterns in the affected modules, then drafts design.md following the SDD-structure (AD-X decisions, file structure, data models, API contracts). A second review pass — by the same agent or a separate Opus instance — checks traceability (every FR-X traces to at least one AD-X), validates that alternatives were genuinely considered, and flags API contracts missing error cases. The human approves before the agent generates the implementation-plan.md.

### Recommended subagents
- `faion-sdd-executor-agent` — consumes design.md as the authoritative source for file structure and patterns during task execution
- Opus-tier subagent — AD-X authoring, alternatives analysis, trade-off matrices; not suitable for Haiku (design decisions need reasoning depth)
- Sonnet-tier subagent — review pass for completeness, traceability, and clarity of API contracts

### Prompt pattern
```
You are writing design.md for feature-NNN-<slug>.
Inputs: spec.md (attached), constitution.md (attached), codebase patterns from <key files>.
Requirements:
- One AD-X entry per major architectural decision
- Each AD-X: context, decision, rationale, alternatives table, consequences
- File structure section: list every file to create or modify with its purpose
- API contracts: OpenAPI-style, include all error codes
- Traceability: every FR-X from spec.md must appear in at least one AD-X
Output design.md following the SDD structure. Flag any FR-X without a clear design decision.
```

```
Review this design.md:
1. Traceability: list FR-X items from spec.md not covered by any AD-X
2. Alternatives: identify AD-X entries with only one option considered (not a real decision)
3. API contracts: flag endpoints missing error responses or with vague field types
4. File structure: flag files referenced but not defined in the codebase or this design
Return a numbered issue list, then propose fixes for each.
```

## Templates & scripts
See `templates.md` for the Google-style and SDD-style design.md templates, plus ADR and RFC variants.

AD-X entry validator (checks required fields are present):
```bash
#!/usr/bin/env bash
# Usage: ./validate-design.sh design.md
set -euo pipefail
FILE=$1
ERRORS=0
# Check every AD-N block has Status, Decision, Rationale
while IFS= read -r line; do
  if [[ $line =~ ^###\ AD-[0-9]+ ]]; then
    BLOCK=$(awk "/$line/{found=1} found && /^###\ AD-[0-9]+/ && NR>1{exit} found{print}" "$FILE")
    for field in "Status:" "Decision:" "Rationale:"; do
      if ! echo "$BLOCK" | grep -q "$field"; then
        echo "MISSING $field in: $line"; ERRORS=$((ERRORS+1))
      fi
    done
  fi
done < "$FILE"
[[ $ERRORS -eq 0 ]] && echo "OK" || exit 1
```

## Best practices
- Write the AD-X alternatives table before writing the decision — the act of listing alternatives surfaces the real trade-offs
- Link every AD-X to the FR-X requirements it satisfies using a `Traces to:` field — traceability enables future debugging without reading the full doc chain
- Include concrete type definitions (TypeScript interfaces, Pydantic models) in design.md — agents generate more accurate code from explicit types than from prose descriptions
- Mark the status of each AD-X (`Proposed` / `Accepted` / `Superseded`) — agents must not implement `Proposed` decisions without human approval
- Keep design.md in the same Git commit as the first task that implements it; stale docs at old commits confuse agents retracing history
- Separate the design doc from the code PRs — design review and code review are different gates requiring different reviewers

## AI-agent gotchas
- LLMs generate symmetric alternatives tables (Option A: pros/cons, Option B: pros/cons) that look balanced but favor the first option they introduced — human must validate that the "winning" option is genuinely better
- Agents skip the `Alternatives Considered` section when under token pressure; require it explicitly in the prompt
- Design docs generated from a single large prompt with spec + codebase context frequently hallucinate file paths — validate every file listed in the file structure section against `ls` or `fd`
- An agent working from design.md alone (without spec.md) cannot verify FR-X coverage; always load both in executor context
- `Superseded` ADRs must not be removed — agents asked to "clean up" docs may delete them, erasing decision history; add a note to the executor prompt to treat ADRs as append-only
